# Ручное тестирование Dating Bot

Чек-лист для защиты проекта и регрессии перед релизом. Отметьте пункты после проверки.

## Окружение

- [ ] `docker compose up -d` в каталоге `project` (Postgres, Redis, RabbitMQ, MinIO). В **`project/.env`** задан **`BOT_TOKEN`** (тот же, что у бота) — иначе `event_consumer` сразу завершится: без токена нельзя слать push о мэтче. Учётные данные инфраструктуры по умолчанию: **`dating_user`** / **`dating_password`** (Postgres на хосте **5433**, база **`dating_db`**; RabbitMQ UI **http://localhost:15672**; MinIO Console **http://localhost:9001**).
- [ ] Запущен **profile_service** (порт `8201`).
- [ ] Контейнер **`event_consumer`** в статусе running (поднимается тем же `docker compose`).
- [ ] Запущен **bot_service** (polling).

## Регистрация и анкета

- [ ] `/start` — пользователь создаётся, нет дублирующей ошибки при повторном `/start`.
- [ ] Заполнение анкеты (имя, био, возраст, пол, город) — данные сохраняются, повторный просмотр профиля показывает введённое.
- [ ] Загрузка фото — файл появляется в MinIO, URL в анкете, превью в боте открывается.

## Поиск и взаимодействия

- [ ] Выдача кандидатов с учётом фильтров (возраст / пол / город в предпочтениях).
- [ ] **Лайк** и **пропуск** сохраняются; повторно тот же кандидат не показывается сразу после пропуска/лайка (логика ленты).
- [ ] При **взаимном лайке** оба пользователя в течение нескольких секунд получают **личное сообщение в Telegram** от бота (через `event_consumer`), текст про мэтч и ссылку на раздел «Мои мэтчи».

## API и интеграции (по желанию через curl / Postman)

- [ ] `GET http://localhost:8201/health` → `200`, JSON `healthy`.
- [ ] `GET http://localhost:8201/metrics` → `200`, текст в формате Prometheus.

## Нефункциональные проверки

- [ ] После перезапуска profile service кэш Redis для ленты пересобирается (или прогревается Celery beat по расписанию).
- [ ] Логи в stdout читаемы (JSON), без необработанных traceback при нормальном сценарии.
